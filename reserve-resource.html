How Open24 reserves a tennis court resource

This page summarizes the Resource Booking Checkout architecture from open24-docs/ARCHITECTURE.md, including the core APIs and fulfillment steps.

1-hour slots Start at :00 or :30 BFF + Stripe + Strapi

Reservation flow chart

1

Select slot(s)

User picks court slots on reserve page. Cart is staged in session storage.

2

Create checkout session

Client calls POST /api/stripe/create-checkout-session on BFF.

3

Process payment

New card via Payment Element or saved-card via charge-saved-card.

4

Fulfill order

Create Booking, ResourceTimeSlots, Payment record, and apply credits/passes if used.

5

Confirmation

Send confirmation and redirect user to success page.

Source: “Checkout Flows → 1. Resource Booking Checkout” in open24-docs/ARCHITECTURE.md.

Key API calls in the booking path

  • POST /api/stripe/create-checkout-session — initialize checkout session from selected slots.
  • POST /api/stripe/charge-saved-card — optional saved-card payment path.
  • Booking fulfillment creates backend records for booking, time slots, and payment linkage.
  • Optional deductions: store credits or pass units, depending on selected payment strategy.

Integrity rules (from architecture)

  • Booking slots are enforced as valid reservation windows and should not overlap.
  • Stripe operations are proxied via BFF server routes for secret-key safety.
  • BFF-to-backend calls use an internal shared secret header for trusted server traffic.
