Customer Journey (Full Path)

1

Selection and staging

Customer selects one or more courses/sessions. Pending enrollment items are staged and user enters checkout screen.

2

Review + waiver

Checkout displays attendee/session details, validates waiver, and applies credits/coupon context before payment.

3

Checkout session creation

The app requests a secure enrollment checkout session based on selected course sessions.

4

Payment execution path

Enrollment supports new-card confirm, saved-card charge, or credits-only path (depending on amount due and selected method).

5

Batch finalization

Batch finalization creates enrollment, payment, and session attendance records, plus waiver linkage for each pending item.

6

Inline confirmation

Checkout shows confirmation directly on-page (no dedicated success route), then user can inspect enrollment records in list/detail pages.

Core System Sequence

Phase System Action Purpose
Checkout init Create secure enrollment checkout session Prepare payment details based on selected sessions.
Saved-card path Charge saved payment method Process payment using a stored card.
Enrollment create Create enrollment records Create an enrollment entity for each pending item.
Attendance create Create attendance records for selected sessions Create session attendance entries for all selected sessions.
Payment + waiver Create payment and waiver records Persist payment audit and waiver linkage for enrollment context.

Enrollment Variants and Notes

Payment variants

  • Full Stripe (new card)
  • Saved card
  • Credits-only (when credit balance fully covers amount)
  • Credits + Stripe mixed path (if remainder exists)

Operational notes

  • Checkout is shared with on-demand mode in the app, but this page focuses on course enrollment flow.
  • Batch finalization behavior is key for multi-course checkout reliability.
  • Enrollment confirmation is rendered inline rather than redirecting to a dedicated success page.
